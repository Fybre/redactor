📄 Overview

Redactor is a self-hosted service that automatically detects and removes personally identifiable information (PII) from PDF documents and images. It supports two ways to submit documents (REST API upload or folder polling) and two ways to receive results (saved to an output directory or delivered via webhook).

Redaction is performed at two levels simultaneously:

  • Visual — PII is covered with solid black boxes in the rendered document
  • Text layer — for text-based PDFs, the underlying text is permanently removed from the PDF content stream so it cannot be extracted by copy/paste or PDF analysis tools
ℹ️
Scanned documents and images are redacted visually using OCR to locate PII. The text layer of scanned PDFs is replaced with the redacted image, so no recoverable text remains.

⚙️ How It Works

Text-based PDFs

For PDFs created digitally (not scanned), Redactor uses PyMuPDF to extract the text layer with precise bounding boxes for every word. Presidio NLP analysis runs on the full page text to identify PII spans. Each detected span is mapped back to its word bounding boxes and a redact annotation is applied.

PyMuPDF's apply_redactions() then permanently removes the characters from the PDF content stream — the black box you see is not just painted on top, the text underneath is gone. Saving with garbage collection removes all orphaned objects from the file.

Scanned PDFs & Images

For scanned documents and images, Redactor renders each page at 300 DPI, runs Tesseract OCR to extract words and their pixel bounding boxes, then maps PII detections back to pixel coordinates and draws filled black rectangles over them. For PDFs, the original page content is replaced with the redacted image.

Mixed documents

Each page in a PDF is assessed individually. Pages with a real text layer use the text pipeline; pages without (scanned pages embedded in a hybrid document) use the OCR pipeline. The output is a single PDF with both types of pages correctly redacted.

PII detection engine

Detection uses Microsoft Presidio backed by the spaCy en_core_web_lg NLP model. Presidio combines rule-based pattern matching (for structured data like credit card numbers and SSNs) with NLP-based named entity recognition (for unstructured text like names and locations).

🛡 Redaction Levels

Each level is a superset of the previous — choosing Aggressive includes everything in Standard, and so on.

⬆️ Uploading Documents

Web UI

The Upload page supports drag-and-drop or click-to-browse for PDF, PNG, JPG, JPEG, TIFF, and BMP files. Multiple files can be queued before submitting — they are uploaded and processed individually.

Options on upload

  • Redaction level — choose from the five levels, or select Custom and tick individual entity types
  • Profile — select a saved profile to pre-populate the custom entity checkboxes
  • Output mode — Save to output directory (default) or Send to webhook (enter a URL)

REST API

Submit via multipart/form-data POST to /api/v1/jobs/upload:

curl -X POST http://localhost:8080/api/v1/jobs/upload \
  -F file=@report.pdf \
  -F level=standard \
  -F output_mode=directory

With custom entity types:

curl -X POST http://localhost:8080/api/v1/jobs/upload \
  -F file=@report.pdf \
  -F level=custom \
  -F 'custom_entities=["PERSON","EMAIL_ADDRESS","PHONE_NUMBER"]' \
  -F output_mode=directory

With webhook delivery:

curl -X POST http://localhost:8080/api/v1/jobs/upload \
  -F file=@report.pdf \
  -F level=aggressive \
  -F output_mode=webhook \
  -F webhook_url=https://your-server.com/hook

Supported formats

  • PDF — text-layer PDFs and scanned/image PDFs, including mixed documents
  • PNG, JPG, JPEG, TIFF, TIF, BMP — redacted using OCR

📁 Folder Polling

When folder polling is enabled, Redactor watches the /data/input directory and automatically submits any new files it finds. This is useful for integrating with other systems that drop files into a shared folder.

How to drop files

From the Docker host, copy files into the container's input directory:

# Using docker cp
docker cp document.pdf redactor-backend:/data/input/

# Using docker exec and a bind-mounted host directory
cp /local/path/document.pdf /host/mount/input/
💡
For production use, mount a host directory into the container so you can write files directly from the host filesystem. See the Admin Reference for how to configure bind mounts.

How polling works

  • The poller runs at the configured interval (default 15 seconds)
  • Each file's SHA-256 hash is recorded — the same file is never processed twice even if re-copied
  • After pickup, files are moved out of /data/input and stored in /data/originals
  • Polled files use the system's default redaction level and default output mode, configurable on the Configuration page

Enabling / disabling polling

Go to Configuration → System Settings and toggle Enable automatic folder polling. Changes take effect at the next poll cycle — no restart required.

📋 Managing Jobs

Job statuses

  • queued — waiting for a worker slot
  • processing — redaction in progress
  • completed — redacted file is ready for download
  • failed — an error occurred; see the job detail page for the error message
  • cancelled — the job was deleted before processing started

Dashboard

The Dashboard shows all jobs with live auto-refresh (every 5 seconds). Use the filter tabs to narrow by status. Click View to see the full job detail, or ⬇ to download a completed job directly.

Job detail page

The job detail page shows:

  • Full job metadata (file name, ID, level, source, processing time)
  • Entity breakdown — a bar chart of how many instances of each entity type were found
  • Full JSON report from the API
  • Download buttons for the redacted file and (if retention is on) the original

Retrying failed jobs

If a job fails (e.g. due to a corrupted file or temporary error), open the job detail page and click ↺ Retry. The job is re-queued with the same settings. You can also retry via the API:

curl -X POST http://localhost:8080/api/v1/jobs/{job_id}/retry

Deleting jobs

Deleting a job removes its database record and both the input and output files. Queued jobs are cancelled instead of immediately deleted. Jobs that are currently processing cannot be deleted.

📋 Custom Redaction Profiles

Profiles let you save a named set of entity types so you don't have to re-select them for every job. A profile is equivalent to the Custom level with a pre-defined entity list.

Creating a profile

1
Open Configuration → Redaction Profiles

Click the + New Profile button.

2
Name the profile and select entities

Give it a descriptive name (e.g. "GDPR Essentials"), optionally add a description, and tick the entity types to include.

3
Save

The profile is immediately available on the Upload page's profile dropdown.

Using a profile on upload

On the Upload page, select Custom level, then choose the profile from the "Or use a saved profile" dropdown. The entity checkboxes are pre-populated from the profile.

Via the API:

curl -X POST http://localhost:8080/api/v1/jobs/upload \
  -F file=@document.pdf \
  -F profile_name="GDPR Essentials"
💡
Set a profile as the system default on the Configuration page so all polled files automatically use it.

🔔 Webhooks

Webhooks let external systems receive a notification as soon as a job completes, including a URL to download the redacted file.

Registering a webhook

Go to Configuration → Webhooks and fill in the URL, an optional display name, and an optional signing secret. Click Add.

Alternatively, via the API:

curl -X POST http://localhost:8080/api/v1/config/webhooks \
  -H "Content-Type: application/json" \
  -d '{"url": "https://your-server.com/hook", "name": "My Server", "secret": "s3cret"}'

Webhook payload

Redactor sends a JSON POST to your endpoint when a job completes:

{
  "event": "job.completed",
  "job_id": "a1b2c3d4-...",
  "filename": "document.pdf",
  "status": "completed",
  "level": "standard",
  "page_count": 3,
  "entities_found": {"PERSON": 4, "EMAIL_ADDRESS": 2},
  "processing_ms": 1842,
  "completed_at": "2026-03-18T10:00:01Z",
  "download_url": "http://redactor-host/api/v1/jobs/a1b2c3d4-.../download"
}

Verifying signatures

If a signing secret is set, each request includes an HMAC-SHA256 signature:

X-Redactor-Signature: sha256=abc123...
X-Redactor-Timestamp: 1710756001

Verify in Python:

import hmac, hashlib

def verify(body: bytes, secret: str, header: str) -> bool:
    expected = hmac.new(secret.encode(), body, hashlib.sha256).hexdigest()
    received = header.removeprefix("sha256=")
    return hmac.compare_digest(expected, received)

Retries

If your endpoint returns a non-2xx response or is unreachable, Redactor retries up to 3 times with exponential backoff (1s, 2s, 4s). After 3 failures the delivery is marked as failed in the job record.

Testing a webhook

On the Configuration → Webhooks tab, click Test next to any registered webhook. A test payload is sent immediately and the result (success or failure) is shown as a notification.

🗂 Therefore™ Integration

This section covers the full round-trip workflow between Therefore and Redactor. There are two approaches — choose the one that fits your category setup.

Approach A — Webhook Template
Redactor posts directly to Therefore's CreateDocument endpoint. No adapter service required.
Therefore ─▶ Redactor ─▶ Therefore CreateDocument
Approach B — Adapter Service
A small service reshapes the payload. Use when you need PreprocessIndexData or dynamic keyword lookups.
Therefore ─▶ Redactor ─▶ Adapter ─▶ Therefore CreateDocument

Approach A — Webhook Template (no adapter)

Step 1A — Create the webhook template

Go to Configuration → Webhook Templates and open the pre-loaded therefore_create_document template (or create a new one). The body is Jinja2 JSON that maps Redactor job variables directly to Therefore's CreateDocument structure:

        "FieldNo": 0,
        "DataValue": "{{ stem }}"
      }
    },
    {
      "StringIndexData": {
        "FieldName": "Redaction_Level",
        "FieldNo": 0,
        "DataValue": "{{ level }}"
      }
    },
    {
      "StringIndexData": {
        "FieldName": "Job_ID",
        "FieldNo": 0,
        "DataValue": "{{ job_id }}"
      }
    },
    {
      "DateIndexData": {
        "FieldName": "Redaction_Date",
        "FieldNo": 0,

Step 3B — The adapter saves the document back to Therefore

import hashlib, hmac, os, requests
from fastapi import FastAPI, Request, HTTPException

app = FastAPI()

REDACTOR_WEBHOOK_SECRET = os.environ["REDACTOR_WEBHOOK_SECRET"]
THEREFORE_BASE_URL      = os.environ["THEREFORE_BASE_URL"]
THEREFORE_USERNAME      = os.environ["THEREFORE_USERNAME"]
THEREFORE_PASSWORD      = os.environ["THEREFORE_PASSWORD"]
THEREFORE_TENANT        = os.environ.get("THEREFORE_TENANT", "")
THEREFORE_CATEGORY_NO   = int(os.environ["THEREFORE_CATEGORY_NO"])


def therefore_session():
    s = requests.Session()
    s.auth = (THEREFORE_USERNAME, THEREFORE_PASSWORD)
    s.headers.update({"Content-Type": "application/json; charset=utf-8"})
    if THEREFORE_TENANT:
        s.headers["TenantName"] = THEREFORE_TENANT
    return s

def therefore_post(session, endpoint, payload):
    resp = session.post(f"{THEREFORE_BASE_URL}/{endpoint}", json=payload, timeout=60)
    resp.raise_for_status()
    return resp.json()

def verify_signature(body: bytes, header: str) -> bool:
    expected = hmac.new(
        REDACTOR_WEBHOOK_SECRET.encode(), body, hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, header.removeprefix("sha256="))


@app.post("/redactor-callback")
async def redactor_callback(request: Request):
    body = await request.body()
    if not verify_signature(body, request.headers.get("X-Redactor-Signature", "")):
        raise HTTPException(status_code=401, detail="Invalid signature")

    payload = await request.json()
    if payload.get("status") != "completed" or not payload.get("file_data"):
        return {"ok": True, "skipped": True}

    session = therefore_session()

    # Build index data — adjust FieldNo values to match your category definition.
    # Use POST /restun/GetCategoryInfo {"CategoryNo": N} to list field numbers.
    index_items = [
        {"StringIndexData": {"FieldNo": 101, "DataValue": payload["filename"]}},
        {"StringIndexData": {"FieldNo": 102, "DataValue": payload["job_id"]}},
    ]

    # PreprocessIndexData — fills defaults, auto-numbering, calculated fields
    preprocessed = therefore_post(session, "PreprocessIndexData", {
        "CategoryNo": THEREFORE_CATEGORY_NO,
        "FillDependentFields": True, "ResetToDefaults": True,
        "DoCalculateFields": True, "GetAutoAppendIxData": False,
        "ExcludeReduntantForFillDependentFields": True,
        "IndexData": {"IndexDataItems": index_items},
    })

    # CreateDocument — file_data is already base64, pass it through directly
    result = therefore_post(session, "CreateDocument", {
        "TheDocument": {
            "CategoryNo": THEREFORE_CATEGORY_NO,
            "IndexDataItems": preprocessed["IndexData"]["IndexDataItems"],
            "Streams": [{
                "StreamNo": 0,
                "FileName": payload["file_name"],
                "FileData": payload["file_data"],
            }],
        }
    })
    return {"ok": True, "doc_no": result.get("DocNo")}

Notes

  • Approach A vs B — use the template approach when index data is simple or static. Use the adapter when you need PreprocessIndexData, calculated fields, or dynamic keyword lookups.
  • Field numbers — in the template or adapter, set FieldNo to match your Therefore category definition. Look them up with:
    POST /restun/GetCategoryInfo {"CategoryNo": 8, "IsAccessMaskNeeded": false}
  • Keyword fields — if the category has mandatory keyword (dropdown) fields, resolve their display values to KeywordNo integers using GetKeywordsByFieldNo. This usually requires the adapter approach.
  • Therefore Online — include "TenantName":"acme" in webhook_headers (Approach A) or set THEREFORE_TENANT in the adapter env (Approach B).
  • Redaction level per document — Therefore's outgoing REST call can substitute index field values into form parts, so you can map a Therefore field to the level form part to vary redaction level per document.
⚠️
Large files — base64 inflates file size by ~33%. For documents over ~20 MB, consider setting webhook_include_file=false and fetching the file separately from /api/v1/jobs/{job_id}/download using the download_url in the standard payload.

    driver_opts:
      type: none
      o: bind
      device: /srv/redactor/data  # ← your host path

Adding authentication

The API and web UI have no built-in authentication. Before exposing Redactor to a network, put it behind a reverse proxy with authentication:

  • Caddy — basicauth directive or forward_auth for OAuth
  • nginx — auth_basic or auth_request for OAuth
  • Traefik — BasicAuth or ForwardAuth middleware

nginx client_max_body_size

The nginx config sets client_max_body_size 110M. If you increase MAX_FILE_SIZE_MB beyond 100, update this value in frontend/nginx.conf and rebuild the frontend image:

docker compose build frontend
docker compose up -d frontend

Resetting all data

🚨
This permanently deletes all jobs, redacted files, and configuration.
docker compose down
docker volume rm redactor_redactor-data
docker compose up -d
