📄 Overview

Redactor is a self-hosted service that automatically detects and permanently removes personally identifiable information (PII) from documents. It supports PDFs (both text-based and scanned), PNG, JPG, and TIFF images.

Documents can be submitted via the web UI, the REST API, or by dropping files into a watched input folder. The redacted output can be saved to a directory or delivered to any HTTP endpoint via a signed webhook — including directly to third-party document management systems like Therefore.

Detection engine

Redactor uses pattern matching and statistical NLP — not a generative AI model. There is no external API, no internet dependency, and no data leaves your server.

ComponentRole
Microsoft PresidioOrchestration layer; runs recognisers and returns entity spans
spaCy en_core_web_lgNamed Entity Recognition for persons, organisations, locations — a statistical ML classifier trained on annotated text (not generative AI)
Presidio regex recognisersStructured identifiers: credit cards, SSNs, IBANs, phone numbers, emails, passports, and regional ID types
Tesseract OCRExtracts text from scanned pages — pure computer vision
PyMuPDFText extraction and PDF manipulation

Detection is fast and deterministic. It can miss PII that doesn't match known patterns, and may occasionally produce false positives on ambiguous text.

⚙️ How It Works

Text-based PDFs

For PDFs that contain a real text layer (digitally created, not scanned), Redactor extracts words with exact bounding boxes, runs PII detection on the full text, then uses PyMuPDF's apply_redactions() to permanently remove the text from the PDF content stream. The redacted text cannot be recovered by selecting, copying, or forensic PDF analysis tools.

Scanned PDFs & Images

For image-based PDFs and standalone images, Redactor renders each page at 300 DPI, runs Tesseract OCR to extract words with pixel bounding boxes, detects PII in the OCR text, then draws filled black rectangles over the identified regions. For scanned PDFs, the redacted image replaces the original page content.

Mixed documents

Each page is assessed individually. Pages with a real text layer use the text pipeline; image-only pages use the OCR pipeline. Both can appear in a single output PDF.

Forensic integrity

✅
For text-layer PDFs, redacted content is removed from the content stream — not just painted over. The output file passes forensic inspection; the original text is genuinely absent.

⬆️ Uploading Documents

Web UI

Go to the Upload page. Select your file, choose a redaction level (or a saved profile), and set an output mode:

  • Save to output directory — the redacted file is saved to /data/output/
  • Send to webhook — the redacted file is delivered to an HTTP endpoint on completion

When webhook output is selected, additional fields appear for the endpoint URL, optional template, and whether to embed the file as base64 in the payload.

REST API

Submit a multipart POST to /api/v1/jobs/upload:

curl -X POST http://redactor-host:8080/api/v1/jobs/upload \
  -F file=@invoice.pdf \
  -F level=standard \
  -F output_mode=directory

Webhook output with a template:

curl -X POST http://redactor-host:8080/api/v1/jobs/upload \
  -F file=@invoice.pdf \
  -F level=standard \
  -F output_mode=webhook \
  -F webhook_url=https://dms.example.com/api/CreateDocument \
  -F webhook_template=therefore_create_document \
  -F webhook_include_file=true \
  -F 'webhook_extra={"category_no": 299}'

📁 Folder Polling

Redactor can watch one or more input folders and automatically process any file that appears in them. No API call required — drop a file and it appears as a completed job on the Dashboard within one poll interval.

Watched folders

Go to Configuration → Watched Folders to manage input folders. Each folder has:

  • Name — a label for the dashboard
  • Input path — absolute path inside the container (e.g. /data/input/hr)
  • Output path — where redacted files are saved (leave blank to use the global output directory)
  • Redaction profile — which profile to apply to files from this folder (leave blank to use the system default level)
  • Enabled — toggle a folder on or off without deleting it
💡
Use sub-folders to route different document types to different redaction profiles. For example: /data/input/hr with a "HR Documents" profile, /data/input/finance with a "Finance Minimal" profile, each writing to its own output path.

Example: three watched folders with different profiles

FolderInput pathProfileOutput path
HR Documents/data/input/hrHR Standard/data/output/hr
Finance/data/input/financeminimal/data/output/finance
Legal/data/input/legalmaximum/data/output/legal

How to drop files

# Via docker cp (named volume)
docker cp document.pdf redactor-backend:/data/input/hr/

# Via host-mounted volume
cp document.pdf /your/host/path/input/hr/

How polling works

  • Each file's SHA-256 hash is recorded on first processing — the same file is never processed twice
  • Files are moved out of the input folder after being queued to prevent re-processing
  • macOS resource fork files (._filename) and other hidden dot-files are silently deleted and ignored
  • The poll interval is configurable (default 15 s) and can be changed without restarting

Enabling / disabling polling

Go to Configuration → System Settings and toggle Enable automatic folder polling. Individual folders can also be toggled on/off in Configuration → Watched Folders.

📋 Redaction Profiles

A redaction profile is a saved, named combination of entity types and a detection strategy. Profiles let you define compliance-specific configurations once and reuse them across watched folders, API calls, and the upload UI.

Built-in profiles

The four standard redaction levels — minimal, standard, aggressive, and maximum — appear automatically as built-in profiles in Configuration → Redaction Profiles. They cannot be edited directly, but you can duplicate them and customise the copy.

Click Restore defaults to bring back any built-in profile that was deleted.

Creating a custom profile

Go to Configuration → Redaction Profiles → New Profile. Give it a name, an optional description, choose a detection strategy, and tick the entity types to include.

Via API:

curl -X POST http://redactor-host:8080/api/v1/config/profiles \
  -H "Content-Type: application/json" \
  -d '{
    "name": "GDPR Essentials",
    "entities": ["PERSON", "EMAIL_ADDRESS", "PHONE_NUMBER", "LOCATION"],
    "description": "Core GDPR personal data fields",
    "strategy": "both"
  }'

Per-profile detection strategy

Each profile can specify its own detection strategy, overriding the system default for jobs that use that profile:

StrategyWhen to use
Inherit system defaultNo override — use whatever the system setting is. Default for all profiles.
presidioPattern matching and NER only. Fast and deterministic. Best for structured data (IDs, emails, credit cards).
llmLLM detection only. Better for contextual PII in free-form text (e.g. "my address is…"). Requires Ollama to be configured.
bothRun Presidio and LLM in parallel and merge results. Most thorough; LLM spans that overlap >80% with a Presidio span are deduplicated.
⚠️
A profile's strategy must be compatible with the system-level detection strategy setting. If the system is set to presidio only and a profile requests llm, the job falls back to the system strategy and a warning is logged. The Detection Strategy column in the profile list shows an ⚠ indicator for incompatible profiles.

Example profiles

HR Documents — full person data with LLM for contextual names:

{
  "name": "HR Documents",
  "entities": ["PERSON", "EMAIL_ADDRESS", "PHONE_NUMBER", "DATE_TIME",
               "LOCATION", "NRP", "MEDICAL_LICENSE"],
  "description": "Full HR redaction — names, contact details, dates, NI numbers",
  "strategy": "both"
}

Finance Minimal — structured identifiers only, Presidio is sufficient:

{
  "name": "Finance Minimal",
  "entities": ["CREDIT_CARD", "IBAN_CODE", "BANK_ACCOUNT", "UK_NHS"],
  "description": "Financial identifiers only — no names or addresses",
  "strategy": "presidio"
}

Using a profile on upload

curl -X POST http://redactor-host:8080/api/v1/jobs/upload \
  -F file=@document.pdf \
  -F profile_name="GDPR Essentials"

Duplicating a profile

Click Duplicate on any profile to create an editable copy. This is the recommended way to customise a built-in level without losing the original.

🔍 Custom Entities

The Configuration → Entities tab shows every entity type the system can detect, with its type (pattern/NER/deny-list) and whether it is a custom or built-in recogniser. You can extend detection with your own pattern-based or deny-list recognisers.

How detection works

ModeHow it finds PIIBest for
Presidio Regex patterns + spaCy NER. Fully deterministic — same input always produces the same result. Structured identifiers (emails, credit cards, SSNs, phone numbers, IBANs) and named entities (persons, locations) in well-formed text.
LLM Sends the text to a local Ollama model with a structured prompt listing the entity types and their descriptions. The model returns spans with type labels. Free-form prose where PII is written contextually rather than in a predictable format. Also the only way to detect custom entity types that have no fixed pattern.
Both Runs Presidio and LLM in parallel. Merges results; LLM spans that overlap >80% with a Presidio span are dropped (Presidio is more precise on structured data). Documents that mix structured and free-form PII. Highest recall at the cost of some extra processing time.

Adding a pattern recogniser

Pattern recognisers detect values that match a regular expression — ideal for internal identifiers with a fixed format (employee IDs, case numbers, account codes).

Go to Configuration → Entities → Add Custom Recogniser, select Pattern (regex), and fill in:

  • Name — a display label (e.g. "Employee ID")
  • Entity Type — the label applied to matches (e.g. EMPLOYEE_ID). Use an existing type to extend it, or a new name to create a new category.
  • Regex Pattern — the regular expression (e.g. EMP-\d{5})
  • Confidence Score — 0.0–1.0. Controls the minimum score reported; higher values mean the match must look more certain before being flagged. Default 0.5 is a safe starting point.
  • Context Words — optional comma-separated words that boost confidence when they appear near the match (e.g. employee, staff, id)
  • Description — a plain-English explanation used in the LLM prompt so the model can find the entity contextually even when it doesn't match the regex (e.g. "An internal employee identifier in the format EMP-NNNNN")

Via API:

curl -X POST http://redactor-host:8080/api/v1/config/recognizers \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Employee ID",
    "entity_type": "EMPLOYEE_ID",
    "type": "pattern",
    "patterns": [{"name": "emp-id", "regex": "EMP-\\d{5}", "score": 0.85}],
    "context": ["employee", "staff", "id", "number"],
    "description": "An internal employee identifier in the format EMP-NNNNN"
  }'

Adding a deny-list recogniser

Deny-list recognisers match exact strings from a fixed list — useful for internal names, project codes, or other values that aren't PII in the generic sense but should be redacted in your context.

Go to Configuration → Entities → Add Custom Recogniser, select Deny List, and enter one value per line in the Deny List Values field.

Via API:

curl -X POST http://redactor-host:8080/api/v1/config/recognizers \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Internal Project Names",
    "entity_type": "PROJECT_NAME",
    "type": "deny_list",
    "deny_list": ["Project Phoenix", "Project Atlas", "Operation Clearwater"],
    "context": ["project", "operation", "codename"],
    "description": "Internal project codenames that must not appear in externally shared documents"
  }'
💡
LLM and custom entities: The Description field is passed directly to the LLM prompt when using the llm or both detection strategy. A clear, specific description (e.g. "A 10-digit internal case reference beginning with CR-") dramatically improves the model's ability to find instances that don't exactly match your regex — for example, when the value is paraphrased or partially obscured.

Using custom entities in a profile

Once a custom recogniser is added, its entity type appears in the entity list when creating or editing a profile. Tick it alongside any standard entities to include it in the profile's detection set.

For the LLM to detect custom entities, the profile (or system default) must use the llm or both strategy and the entity type must be ticked in the profile.

🔔 Webhooks

When a job's output mode is set to webhook, Redactor posts a JSON payload to your configured endpoint as soon as the job completes. Webhooks are retried up to 3 times with exponential backoff on failure.

Registering a webhook

Go to Configuration → Webhooks → Add Webhook. Enter the URL, an optional name, and an optional signing secret. Registered webhooks receive a notification for every job that completes with output_mode=webhook and that webhook's URL.

Standard webhook payload

{
  "event": "job.completed",
  "job_id": "a1b2c3d4-...",
  "filename": "document.pdf",
  "status": "completed",
  "level": "standard",
  "page_count": 3,
  "entities_found": { "PERSON": 2, "EMAIL_ADDRESS": 1 },
  "processing_ms": 1842,
  "completed_at": "2026-03-18T10:00:01Z",
  "download_url": "http://redactor-host/api/v1/jobs/a1b2c3d4-.../download"
}

If webhook_include_file=true, the payload also includes file_data (base64-encoded redacted file), file_name, and file_size_bytes.

Verifying signatures

If a signing secret is configured, every request includes:

X-Redactor-Signature: sha256=<hmac-hex>
X-Redactor-Timestamp: <unix-timestamp>

Verify in Python:

import hmac, hashlib

def verify(body: bytes, secret: str, sig_header: str) -> bool:
    expected = hmac.new(secret.encode(), body, hashlib.sha256).hexdigest()
    return hmac.compare_digest(expected, sig_header.removeprefix("sha256="))

Retries

Failed webhook deliveries are retried automatically: attempt 1 immediately, attempt 2 after 1 second, attempt 3 after 2 seconds. If all three fail, the job is marked webhook_sent=false.

Testing a webhook

Click Test next to any registered webhook in Configuration → Webhooks to send a test payload immediately.

📝 Webhook Templates

Webhook templates let you control the exact JSON body that Redactor posts to an endpoint. They are Jinja2 templates rendered at job completion time, with the job's data available as variables. This makes it possible to post directly to a third-party API — such as a DMS — without needing an intermediate adapter service.

ℹ️
Three Therefore-ready templates are pre-loaded: therefore_create_document, therefore_update_document, and therefore_update_document2. Edit them to match your category's field definitions.

Date formatting

Some APIs (including Therefore) require dates in yyyy-MM-dd format rather than a full ISO timestamp. Use Jinja2's string slice:

{{ completed_at[:10] }}   {# produces: 2026-03-18 #}

Template headers

Each template has an optional HTTP Headers section — a structured key/value editor where headers are sent with every POST that uses the template. Store API credentials here rather than passing them per-request.

Sensitive headers (those containing words like Authorization, token, key, secret, or password) are automatically masked as you type. Click the 👁 toggle to reveal a value while editing. Non-sensitive headers such as Content-Type or TenantName are shown as plain text.

💡
Storing credentials in the template keeps them out of the job submission request. This is especially useful when Therefore (or another system) is submitting jobs — you don't need to pass auth credentials through the calling system at all.
🔒
Set ALLOW_HEADER_REVEAL=false in .env to prevent sensitive header values from ever being sent to the browser. Once saved, they can only be replaced by typing a new value — they cannot be viewed. Changes to other fields (description, body) can still be saved without re-entering credentials.

Dynamic per-job variables

Pass a webhook_extra JSON object when submitting a job. Every key is merged into the template context and available as a variable:

curl -X POST http://redactor-host:8080/api/v1/jobs/upload \
  -F file=@invoice.pdf \
  -F output_mode=webhook \
  -F webhook_url=https://dms.example.com/CreateDocument \
  -F webhook_template=therefore_create_document \
  -F webhook_include_file=true \
  -F 'webhook_extra={"category_no": 299}'

In the template: {{ category_no | default(57) }} — the default(57) filter provides a fallback when the variable isn't supplied.

Duplicating templates

Click Duplicate on any template to create a copy with all the same headers and body. The copy is named templatename_copy (or _copy_2 etc. if that name is already taken). Useful when you need the same auth credentials but different category numbers or index field mappings across multiple workflows.

Jinja2 tips

  • {{ value | default("fallback") }} — use a fallback if the variable isn't set
  • {{ value | tojson }} — safely encode a value as JSON (useful for the entities_found dict)
  • {% if file_data %}...{% endif %} — conditionally include the file stream only when data is present
  • {{ completed_at[:10] }} — date only (yyyy-MM-dd)

🗂 Therefore™ Integration

Redactor can post directly to Therefore's REST API on job completion using a webhook template — no adapter service required. The recommended pattern is to use the therefore_update_document template to replace an existing document's stream with the redacted version.

⚠️
Therefore REST Call task mangles JSON. The REST Call task in Therefore workflows corrupts JSON string values passed as Body fields — so webhook_headers={"Authorization":"..."} arrives garbled. Use individual prefixed fields instead: one plain-text field per header or extra variable, no JSON required.

Pattern 1 — Immediate update without validation

The simplest flow: Therefore submits a document, Redactor redacts it, and immediately calls UpdateDocument to replace the stream. The REST Call task blocks until the full round-trip is complete.

Store credentials in the webhook template

Go to Configuration → Webhook Templates, edit therefore_update_document, and add your credentials to the HTTP Headers:

{
  "Authorization": "Basic <base64(username:password)>",
  "TenantName": "acme",
  "Content-Type": "application/json"
}

Generate the Basic value: echo -n "username:password" | base64
TenantName is only required for Therefore Online — use your subdomain prefix.

Configure the REST Call task in Therefore

In your Therefore workflow, add a REST Call action targeting /api/v1/jobs/upload-sync. Set these Body fields:

Field nameValue
output_modewebhook
webhook_urlhttps://acme.thereforeonline.com/theservice/v0001/restun/UpdateDocument
webhook_templatetherefore_update_document
webhook_include_filetrue
webhook_extra_doc_no%DocNo% (Therefore workflow variable)
webhook_extra_stream_no0
levelstandard (or your preferred level/profile)

REST Call task URL: POST http://redactor-host:8080/api/v1/jobs/upload-sync

Equivalent curl for testing:

curl -X POST http://redactor-host:8080/api/v1/jobs/upload-sync \
  -F file=@document.pdf \
  -F level=standard \
  -F output_mode=webhook \
  -F webhook_url=https://acme.thereforeonline.com/theservice/v0001/restun/UpdateDocument \
  -F webhook_template=therefore_update_document \
  -F webhook_include_file=true \
  -F webhook_extra_doc_no=4521 \
  -F webhook_extra_stream_no=0
💡
Concurrency token: Therefore's UpdateDocument requires LastChangeTimeISO8601 as a concurrency token — it must match the value currently stored in Therefore, not the time Redactor finished processing. Configure a Pre-fetch URL on the template to fetch the current value immediately before the webhook fires:

Pre-fetch URL: https://acme.thereforeonline.com/theservice/v0001/restun/GetDocumentIndexData
Method: POST
Body: {"DocNo": {{ doc_no }}}

Then in the template body: "LastChangeTimeISO8601": "{{ fetched.IndexData.LastChangeTimeISO8601 }}"

Pattern 2 — Validation workflow (human-in-the-loop)

Use this pattern when a human must review and approve the detected redaction regions before the document is permanently modified. The flow:

  1. Therefore submits the document with validation_mode=true — detection runs but no redaction is applied yet
  2. Redactor returns a validation_url immediately
  3. Therefore stores the URL in a document field and enters a waiting state
  4. A reviewer opens the URL, inspects highlighted regions, approves or rejects them, and adds any missed areas manually
  5. On Save & Apply, Redactor applies only the approved regions, calls UpdateDocument, and fires the completion_callback_url to release the Therefore workflow
Create a validation webhook template

Duplicate therefore_update_document and name it therefore_update_document_validated. Edit its body to also set a logical "Redacted" field and, importantly, use a pre-fetch to get the current LastChangeTimeISO8601 — essential because significant time may pass between submission and the reviewer approving the document:

Pre-fetch settings (on the template):

  • Pre-fetch URL: https://acme.thereforeonline.com/theservice/v0001/restun/GetDocumentIndexData
  • Method: POST
  • Body: {"DocNo": {{ doc_no }}}

Template body:

{
  "DocNo": {{ doc_no }},
  "CheckInComments": "Redacted by Redactor — {{ filename }}",
  "IndexData": {
    "IndexDataItems": [
      {
        "LogicalIndexData": {
          "FieldName": "Redacted",
          "FieldNo": 0,
          "DataValue": true
        }
      }
    ],
    "LastChangeTimeISO8601": "{{ fetched.IndexData.LastChangeTimeISO8601 }}",
    "DoFillDependentFields": false
  },
  "StreamsToUpdate": [
    {% if file_data %}
    {
      "StreamNo": {{ stream_no | default(0) }},
      "FileName": "{{ file_name }}",
      "FileDataBase64JSON": "{{ file_data }}"
    }
    {% endif %}
  ]
}

Set the same Therefore credentials in the HTTP Headers as for the immediate template.

Configure the REST Call task with validation_mode

The REST Call sends the document to Redactor with validation_mode=true and provides a completion_callback_url that Redactor will POST to after the reviewer clicks Save & Apply. This callback is what releases the Therefore workflow from its waiting state.

Set up the Therefore wait condition

After the REST Call task, add a Wait step or condition in the Therefore workflow that pauses until the completion_callback_url is called. Typically this is done by:

  • Setting a logical field (e.g. Pending Review) to true immediately after the REST Call
  • Waiting on that field returning to false
  • Configuring completion_callback_url to point to a Therefore REST endpoint that flips the field back to false

When the reviewer clicks Save & Apply in the validation UI, Redactor fires the completion callback automatically.

Equivalent curl for testing the validation submission:

curl -X POST http://redactor-host:8080/api/v1/jobs/upload-sync \
  -F file=@document.pdf \
  -F level=standard \
  -F validation_mode=true \
  -F output_mode=webhook \
  -F webhook_url=https://acme.thereforeonline.com/theservice/v0001/restun/UpdateDocument \
  -F webhook_template=therefore_update_document_validated \
  -F webhook_include_file=true \
  -F webhook_extra_doc_no=4521 \
  -F completion_callback_url=https://acme.thereforeonline.com/theservice/v0001/restun/SaveDocumentIndexDataQuick

Pattern 3 — Skip review when all regions are auto-approved

When the Auto-approve threshold is set in System Settings (default 0.85), any detected region with a Presidio confidence score at or above that threshold is automatically pre-approved in the validation UI. If all regions are pre-approved — and no manual regions need to be added — the reviewer step can be skipped entirely by calling /apply directly from the Therefore workflow.

Submit with validation_mode=true (same as Pattern 2)

The response includes job_id and region_count.

Check for pending regions

In a second REST Call task, fetch the region list and check whether any are still pending (i.e. not yet approved or rejected):

GET http://redactor-host:8080/api/v1/jobs/{job_id}/regions

The response is a list of region objects each with a status field: auto_approved, pending, approved, or rejected.

In the Therefore workflow, use a condition: if the list contains any item with status = "pending", route to the human review path (Pattern 2). Otherwise, continue to the auto-apply step.

Auto-apply approved regions

If no pending regions exist, call the apply endpoint directly — no human review needed:

POST http://redactor-host:8080/api/v1/jobs/{job_id}/apply

This applies all approved and auto_approved regions, fires the webhook_url (UpdateDocument), and fires the completion_callback_url — exactly the same as when a reviewer clicks Save & Apply.

💡
Tuning auto-approve threshold: Set it higher (e.g. 0.95) to require more human review — only very high-confidence matches are auto-approved. Set it lower (e.g. 0.7) to reduce reviewer workload. All detected regions are always stored; the threshold only controls the initial status assigned to each one.

Notes

  • Prefixed fields vs JSON — use webhook_header_<Name> and webhook_extra_<key> prefixed fields in Therefore's REST Call task body. Both avoid JSON mangling. Prefix fields take precedence when the same key is supplied in both formats.
  • Sync vs async — use /upload-sync when Therefore must wait for the full redact+deliver round-trip (UpdateDocument). Use /upload for fire-and-forget delivery (CreateDocument, archival).
  • Concurrency token — UpdateDocument requires a current LastChangeTimeISO8601. Always configure a Pre-fetch URL on the webhook template so it is fetched immediately before the webhook fires, not at job submission time.
  • Field numbers — FieldNo in index data must match your Therefore category. Use POST /restun/GetCategoryInfo {"CategoryNo": N, "IsAccessMaskNeeded": false} to list them.
  • Keyword fields — if the category has mandatory keyword (dropdown) fields, values must be resolved to KeywordNo integers via GetKeywordsByFieldNo. This typically requires an adapter service rather than a direct template call.
  • Large files — base64 adds ~33% to file size. For documents over ~20 MB, consider webhook_include_file=false and fetching via the download_url in the standard payload.
  • TenantName header — required for Therefore Online only. Omit for on-premises installations.
  • Logical field data type — use LogicalIndexData with "DataValue": true (JSON boolean, no quotes) to set a yes/no field.

Persisting data to the host filesystem

By default Redactor uses a named Docker volume. To use a host-mounted path instead, edit docker-compose.yml:

volumes:
  redactor-data:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: /srv/redactor/data

Adding authentication

The API and web UI have no authentication by default. Before exposing Redactor to a network, put a reverse proxy in front with basic auth or OAuth. Example with Caddy:

redactor.example.com {
    basicauth /* {
        admin JDJhJDE0J...
    }
    reverse_proxy localhost:8080
}
