📄 Overview

Redactor is a self-hosted service that automatically detects and permanently removes personally identifiable information (PII) from documents. It supports PDFs (both text-based and scanned), PNG, JPG, and TIFF images.

Documents can be submitted via the web UI, the REST API, or by dropping files into a watched input folder. The redacted output can be saved to a directory or delivered to any HTTP endpoint via a signed webhook — including directly to third-party document management systems like Therefore.

Detection engine

Redactor uses pattern matching and statistical NLP — not a generative AI model. There is no external API, no internet dependency, and no data leaves your server.

ComponentRole
Microsoft PresidioOrchestration layer; runs recognisers and returns entity spans
spaCy en_core_web_lgNamed Entity Recognition for persons, organisations, locations — a statistical ML classifier trained on annotated text (not generative AI)
Presidio regex recognisersStructured identifiers: credit cards, SSNs, IBANs, phone numbers, emails, passports, and regional ID types
Tesseract OCRExtracts text from scanned pages — pure computer vision
PyMuPDFText extraction and PDF manipulation

Detection is fast and deterministic. It can miss PII that doesn't match known patterns, and may occasionally produce false positives on ambiguous text.

Redaction levels

Standard covers names, emails, phone numbers, and financial identifiers — the right choice for most general-purpose redaction. See the Redaction Levels section for the full breakdown of what each level includes.

⚙️ How It Works

Text-based PDFs

For PDFs that contain a real text layer (digitally created, not scanned), Redactor extracts words with exact bounding boxes, runs PII detection on the full text, then uses PyMuPDF's apply_redactions() to permanently remove the text from the PDF content stream. The redacted text cannot be recovered by selecting, copying, or forensic PDF analysis tools.

Scanned PDFs & Images

For image-based PDFs and standalone images, Redactor renders each page at 300 DPI, runs Tesseract OCR to extract words with pixel bounding boxes, detects PII in the OCR text, then draws filled black rectangles over the identified regions. For scanned PDFs, the redacted image replaces the original page content.

Mixed documents

Each page is assessed individually. Pages with a real text layer use the text pipeline; image-only pages use the OCR pipeline. Both can appear in a single output PDF.

Forensic integrity

✅
For text-layer PDFs, redacted content is removed from the content stream — not just painted over. The output file passes forensic inspection; the original text is genuinely absent.

⬆️ Uploading Documents

Web UI

Go to the Upload page. Select your file, choose a redaction level (or a saved profile), and set an output mode:

  • Save to output directory — the redacted file is saved to /data/output/
  • Send to webhook — the redacted file is delivered to an HTTP endpoint on completion

When webhook output is selected, additional fields appear for the endpoint URL, optional template, and whether to embed the file as base64 in the payload.

REST API

Submit a multipart POST to /api/v1/jobs/upload:

curl -X POST http://redactor-host:8080/api/v1/jobs/upload \
  -F file=@invoice.pdf \
  -F level=standard \
  -F output_mode=directory

Webhook output with a template:

curl -X POST http://redactor-host:8080/api/v1/jobs/upload \
  -F file=@invoice.pdf \
  -F level=standard \
  -F output_mode=webhook \
  -F webhook_url=https://dms.example.com/api/CreateDocument \
  -F webhook_template=therefore_create_document \
  -F webhook_include_file=true \
  -F 'webhook_extra={"category_no": 299}'

📁 Folder Polling

When folder polling is enabled, Redactor automatically processes any file placed in the /data/input/ directory. No API call required — just drop the file and it will appear as a completed job on the Dashboard within one poll interval.

How to drop files

# Via docker cp (named volume)
docker cp document.pdf redactor-backend:/data/input/

# Via host-mounted volume
cp document.pdf /your/host/path/input/

How polling works

  • Each file's SHA-256 hash is recorded on first processing so the same file is never processed twice
  • Files are moved out of /data/input/ after being queued to prevent re-processing
  • Polled files use the system default redaction level and output mode set in Configuration
  • The poll interval is configurable (default 15 seconds) and can be changed without restarting

Enabling / disabling polling

Go to Configuration → System Settings and toggle Enable automatic folder polling. Changes take effect immediately without a restart.

📋 Custom Redaction Profiles

Profiles let you save a named set of entity types so you don't have to specify them individually on every upload. Useful when you have a fixed set of types for a particular workflow or compliance requirement.

Creating a profile

Go to Configuration → Redaction Profiles → New Profile. Give the profile a name, optionally a description, and check the entity types to include.

Via API:

curl -X POST http://redactor-host:8080/api/v1/config/profiles \
  -H "Content-Type: application/json" \
  -d '{
    "name": "GDPR Essentials",
    "entities": ["PERSON", "EMAIL_ADDRESS", "PHONE_NUMBER", "LOCATION"],
    "description": "Core GDPR personal data fields"
  }'

Using a profile on upload

Select the profile from the dropdown on the Upload page, or pass profile_name in the API:

curl -X POST http://redactor-host:8080/api/v1/jobs/upload \
  -F file=@document.pdf \
  -F profile_name="GDPR Essentials"

🔔 Webhooks

When a job's output mode is set to webhook, Redactor posts a JSON payload to your configured endpoint as soon as the job completes. Webhooks are retried up to 3 times with exponential backoff on failure.

Registering a webhook

Go to Configuration → Webhooks → Add Webhook. Enter the URL, an optional name, and an optional signing secret. Registered webhooks receive a notification for every job that completes with output_mode=webhook and that webhook's URL.

Standard webhook payload

{
  "event": "job.completed",
  "job_id": "a1b2c3d4-...",
  "filename": "document.pdf",
  "status": "completed",
  "level": "standard",
  "page_count": 3,
  "entities_found": { "PERSON": 2, "EMAIL_ADDRESS": 1 },
  "processing_ms": 1842,
  "completed_at": "2026-03-18T10:00:01Z",
  "download_url": "http://redactor-host/api/v1/jobs/a1b2c3d4-.../download"
}

If webhook_include_file=true, the payload also includes file_data (base64-encoded redacted file), file_name, and file_size_bytes.

Verifying signatures

If a signing secret is configured, every request includes:

X-Redactor-Signature: sha256=<hmac-hex>
X-Redactor-Timestamp: <unix-timestamp>

Verify in Python:

import hmac, hashlib

def verify(body: bytes, secret: str, sig_header: str) -> bool:
    expected = hmac.new(secret.encode(), body, hashlib.sha256).hexdigest()
    return hmac.compare_digest(expected, sig_header.removeprefix("sha256="))

Retries

Failed webhook deliveries are retried automatically: attempt 1 immediately, attempt 2 after 1 second, attempt 3 after 2 seconds. If all three fail, the job is marked webhook_sent=false.

Testing a webhook

Click Test next to any registered webhook in Configuration → Webhooks to send a test payload immediately.

📝 Webhook Templates

Webhook templates let you control the exact JSON body that Redactor posts to an endpoint. They are Jinja2 templates rendered at job completion time, with the job's data available as variables. This makes it possible to post directly to a third-party API — such as a DMS — without needing an intermediate adapter service.

ℹ️
Three Therefore-ready templates are pre-loaded: therefore_create_document, therefore_update_document, and therefore_update_document2. Edit them to match your category's field definitions.

Template variables

VariableDescription
job_idJob UUID
filenameOriginal filename (with extension)
stemFilename without extension
statuscompleted or failed
levelRedaction level: standard, aggressive, etc.
page_countNumber of pages processed
entities_foundDict of {"ENTITY_TYPE": count}
total_entitiesSum of all entity counts
processing_msProcessing time in milliseconds
completed_atISO 8601 timestamp: 2026-03-18T16:22:14Z
file_dataBase64-encoded redacted file (requires webhook_include_file=true)
file_nameOutput filename
file_size_bytesOutput file size in bytes
any key from webhook_extraMerged in from the per-job webhook_extra JSON object

Date formatting

Some APIs (including Therefore) require dates in yyyy-MM-dd format rather than a full ISO timestamp. Use Jinja2's string slice:

{{ completed_at[:10] }}   {# produces: 2026-03-18 #}

Template headers

Each template has an optional HTTP Headers section — a structured key/value editor where headers are sent with every POST that uses the template. Store API credentials here rather than passing them per-request.

Sensitive headers (those containing words like Authorization, token, key, secret, or password) are automatically masked as you type. Click the 👁 toggle to reveal a value while editing. Non-sensitive headers such as Content-Type or TenantName are shown as plain text.

💡
Storing credentials in the template keeps them out of the job submission request. This is especially useful when Therefore (or another system) is submitting jobs — you don't need to pass auth credentials through the calling system at all.
🔒
Set ALLOW_HEADER_REVEAL=false in .env to prevent sensitive header values from ever being sent to the browser. Once saved, they can only be replaced by typing a new value — they cannot be viewed. Changes to other fields (description, body) can still be saved without re-entering credentials.

Dynamic per-job variables

Pass a webhook_extra JSON object when submitting a job. Every key is merged into the template context and available as a variable:

curl -X POST http://redactor-host:8080/api/v1/jobs/upload \
  -F file=@invoice.pdf \
  -F output_mode=webhook \
  -F webhook_url=https://dms.example.com/CreateDocument \
  -F webhook_template=therefore_create_document \
  -F webhook_include_file=true \
  -F 'webhook_extra={"category_no": 299}'

In the template: {{ category_no | default(57) }} — the default(57) filter provides a fallback when the variable isn't supplied.

Duplicating templates

Click Duplicate on any template to create a copy with all the same headers and body. The copy is named templatename_copy (or _copy_2 etc. if that name is already taken). Useful when you need the same auth credentials but different category numbers or index field mappings across multiple workflows.

Jinja2 tips

  • {{ value | default("fallback") }} — use a fallback if the variable isn't set
  • {{ value | tojson }} — safely encode a value as JSON (useful for the entities_found dict)
  • {% if file_data %}...{% endif %} — conditionally include the file stream only when data is present
  • {{ completed_at[:10] }} — date only (yyyy-MM-dd)

🗂 Therefore™ Integration

This section covers the full round-trip workflow between Therefore and Redactor. The recommended approach posts directly to Therefore's REST API using a webhook template — no adapter service is required.

ℹ️
Flow overview: Therefore sends a document to Redactor via a REST Request action → Redactor redacts it → Redactor renders the webhook template and posts directly to Therefore's CreateDocument endpoint with the redacted file as a base64 stream.

Step 1 — Store credentials in the webhook template

Go to Configuration → Webhook Templates and click Edit on therefore_create_document. In the HTTP Headers field, enter your Therefore credentials as a JSON object:

{
  "Authorization": "Basic <base64(username:password)>",
  "TenantName": "yourtenantname",
  "Content-Type": "application/json"
}
💡
To generate the Basic auth value: echo -n "username:password" | base64
TenantName is only required for Therefore Online (*.thereforeonline.com). Set it to your subdomain prefix — e.g. for acme.thereforeonline.com, use acme.

Step 2 — Update the template body for your category

The default template body covers the most common index fields. Edit it to match your Therefore category's field definitions. Key points:

  • Set FieldNo values to match your category. Look them up with POST /restun/GetCategoryInfo {"CategoryNo": N}.
  • Use {{ completed_at[:10] }} for date fields — Therefore's DateIndexData expects yyyy-MM-dd format only.
  • Use {{ category_no | default(57) }} for the category number so it can be overridden per-job.
{
  "CategoryNo": {{ category_no | default(57) }},
  "CheckInComments": "Redacted by Redactor — {{ filename }}",
  "IndexDataItems": [
    {
      "StringIndexData": {
        "FieldName": "Document_Name",
        "FieldNo": 0,
        "DataValue": "{{ stem }}"
      }
    },
    {
      "DateIndexData": {
        "FieldName": "Redaction_Date",
        "FieldNo": 0,
        "DataISO8601Value": "{{ completed_at[:10] }}"
      }
    }
  ],
  "Streams": [
    {% if file_data %}
    {
      "FileName": "{{ file_name }}",
      "FileDataBase64JSON": "{{ file_data }}",
      "NewStreamInsertMode": 0
    }
    {% endif %}
  ],
  "DoFillDependentFields": true
}

Step 3 — Configure Therefore's outgoing REST Request

In Therefore, create a REST Request workflow action targeting Redactor's upload endpoint. In the Body tab, add the following fields:

Field nameValue
levelstandard
output_modewebhook
webhook_urlYour Therefore CreateDocument URL, e.g. https://acme.thereforeonline.com/theservice/v0001/restun/CreateDocument
webhook_templatetherefore_create_document
webhook_include_filetrue
webhook_extra{"category_no": 57} (optional — omit to use the template's default)
💡
Multiple categories: Duplicate the template for each category and change the default category number in each copy, or pass webhook_extra={"category_no": N} per-job from Therefore to use a single template across multiple categories.

The Therefore REST Request URL should point to Redactor's upload endpoint:

POST http://redactor-host:8080/api/v1/jobs/upload

Equivalent curl for testing the full flow:

curl -X POST http://redactor-host:8080/api/v1/jobs/upload \
  -F file=@invoice.pdf \
  -F level=standard \
  -F output_mode=webhook \
  -F webhook_url=https://acme.thereforeonline.com/theservice/v0001/restun/CreateDocument \
  -F webhook_template=therefore_create_document \
  -F webhook_include_file=true \
  -F 'webhook_extra={"category_no": 57}'

Adapter approach (advanced)

Use an adapter service instead of the direct template approach when you need PreprocessIndexData (auto-numbering, calculated fields, mandatory keyword lookups) before calling CreateDocument. The adapter receives Redactor's standard webhook payload and handles the Therefore API calls itself.

import hashlib, hmac, os, requests
from fastapi import FastAPI, Request, HTTPException

app = FastAPI()

REDACTOR_SECRET  = os.environ["REDACTOR_WEBHOOK_SECRET"]
THEREFORE_BASE   = os.environ["THEREFORE_BASE_URL"]
CATEGORY_NO      = int(os.environ["THEREFORE_CATEGORY_NO"])
THEREFORE_TENANT = os.environ.get("THEREFORE_TENANT", "")


def session():
    s = requests.Session()
    s.auth = (os.environ["THEREFORE_USERNAME"], os.environ["THEREFORE_PASSWORD"])
    s.headers["Content-Type"] = "application/json"
    if THEREFORE_TENANT:
        s.headers["TenantName"] = THEREFORE_TENANT
    return s


@app.post("/redactor-callback")
async def callback(request: Request):
    body = await request.body()
    sig = request.headers.get("X-Redactor-Signature", "")
    expected = hmac.new(REDACTOR_SECRET.encode(), body, hashlib.sha256).hexdigest()
    if not hmac.compare_digest(expected, sig.removeprefix("sha256=")):
        raise HTTPException(401)

    data = await request.json()
    if data.get("status") != "completed" or not data.get("file_data"):
        return {"ok": True}

    s = session()
    result = s.post(f"{THEREFORE_BASE}/CreateDocument", json={
        "CategoryNo": CATEGORY_NO,
        "IndexDataItems": [
            {"StringIndexData": {"FieldNo": 101, "DataValue": data["filename"]}},
        ],
        "Streams": [{
            "StreamNo": 0,
            "FileName": data["file_name"],
            "FileData": data["file_data"],
        }],
    }, timeout=60).json()
    return {"ok": True, "doc_no": result.get("DocNo")}

Notes

  • Keyword fields — if the category has mandatory keyword (dropdown) fields, resolve their values to KeywordNo integers using GetKeywordsByFieldNo before calling CreateDocument. This typically requires the adapter approach.
  • Field numbers — FieldNo must match your Therefore category definition exactly. Use POST /restun/GetCategoryInfo {"CategoryNo": N, "IsAccessMaskNeeded": false} to list field numbers.
  • Large files — base64 encoding adds ~33% to file size. For documents over ~20 MB, consider webhook_include_file=false and fetching via the download_url in the standard payload.
  • TenantName — required for Therefore Online only. Omit for on-premises installations.
  • Testing — submit a test job via the Upload page with output_mode=webhook and check the Docker logs: docker compose logs -f backend. The full request body sent to Therefore is logged (with base64 file data replaced by a size marker).

}
